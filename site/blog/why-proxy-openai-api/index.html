Why Proxying the API is a Good Basis for a Control System

March 2025 · Scott Wofford · Luthien
TL;DR
  • Proxy-over-API as control: Inserting a control proxy between AI agents and LLM APIs provides a powerful first layer of AI oversight. The proxy intercepts all requests and responses in real time, enforces policies, and logs activity—capabilities that alternatives (code wrappers, fine-tuning, in-agent checks) struggle to match.
  • Practical advantages: Full visibility into an AI's I/O, real-time monitoring and intervention, easy policy updates, and modular integration with existing systems. No model retraining required—adoption is frictionless.
  • Alignment incentives: As the first control layer in production, a proxy catches misbehavior immediately, creating a feedback loop that incentivizes safe behavior. Low “alignment tax”—typically just milliseconds of latency.

The Problem

AI agents are getting more capable and more autonomous. Claude Code, GPT-based agents, and open-source frameworks can now write code, browse the web, execute shell commands, and modify your file system. The question isn't whether these agents will do something unexpected—it's when, and whether you'll catch it.

Existing approaches to AI control each have significant limitations. We believe proxying the API—inserting an intermediary between the AI agent and the LLM provider—offers the best foundation for a practical control system.

Proxy vs. Alternative Approaches

1. Code Wrappers & Library Hooks

You could wrap the LLM client library in your codebase to intercept calls. This works for applications you control, but:

  • Fragile: Tightly coupled to specific libraries and versions. SDK updates can break your hooks.
  • Inconsistent: Each agent framework (LangChain, CrewAI, Claude Code, etc.) has different abstractions. A wrapper for one doesn't work for another.
  • Bypassable: Any code path that hits the API directly—a raw HTTP call, a different SDK, a subprocess—skips your controls entirely.

A proxy operates at the network layer, so it catches every API call regardless of how it was made.

2. Fine-Tuning and Model Alignment

Training the model to behave well is important but insufficient:

  • Not available to most users: You can't fine-tune Claude or GPT-4 for custom safety behaviors. You're relying on the provider's alignment work.
  • Alignment is probabilistic: Even well-aligned models produce harmful, incorrect, or off-policy outputs. RLHF and constitutional AI reduce the probability but don't eliminate it.
  • No runtime enforcement: Model-level alignment can't adapt to context-specific rules. "Don't delete production files" or "Always use the staging database" are deployment-specific policies that the model wasn't trained on.

A proxy provides deterministic enforcement of your specific rules, complementing the model's probabilistic alignment.

3. Embedding Controls Inside Agents

You could build safety checks into the agent's own logic:

  • Fox guarding the henhouse: If the agent is the one checking itself, a sufficiently capable (or misaligned) agent could circumvent its own safety checks.
  • Not portable: Controls built into one agent don't transfer to another.
  • Maintenance burden: Every agent needs its own safety implementation, duplicating effort and increasing the surface area for bugs.

An external proxy enforces rules on the agent rather than within it—the control plane is architecturally separated from the thing being controlled.

The key advantages of the proxy approach:

  • Complete visibility: Every request and response flows through the proxy. You see exactly what the agent asked for and what the model returned, including tool calls, system prompts, and multi-turn context.
  • Real-time intervention: The proxy can modify, delay, or block responses before they reach the agent. This isn't after-the-fact logging—it's prevention.
  • Agent-agnostic: Any tool that speaks the OpenAI or Anthropic API protocol works with the proxy. Claude Code, Cursor, Aider, custom agents—one control layer covers all of them.
  • Low alignment tax: Proxying adds ~5–15ms of latency. For a typical LLM call that takes 2–30 seconds, this is imperceptible.
  • Policy flexibility: Rules can be updated without redeploying agents or retraining models. Add a new policy, and it applies to the next API call.

The Proxy as a First Control Layer

AI control is not a single technology—it's a layered strategy. The proxy serves as the first layer: the one closest to the AI's actual behavior in production.

Think of it like network security. You don't rely solely on application-level security. You also have firewalls, IDS/IPS, and network segmentation. The proxy is the firewall for AI agents—it doesn't replace model alignment or careful prompt engineering, but it provides a critical safety net that catches what those approaches miss.

This architecture creates a useful feedback loop:

  1. Log everything: The proxy captures the full interaction history.
  2. Detect violations: Policy checks identify when the agent does something wrong or suspicious.
  3. Intervene in real-time: Block dangerous responses, inject corrections, or alert the user.
  4. Learn from incidents: Logged violations inform new policies, closing gaps over time.

Extensibility

A proxy starts as a monitoring and enforcement layer, but its position in the architecture makes it a natural platform for more sophisticated control:

  • Judge models: Route each response through a separate "judge" model (e.g., Claude Haiku) that evaluates whether the response follows your rules. This happens in parallel to minimize latency.
  • Conversation memory: The proxy maintains the full conversation history, enabling context-aware policies that consider what the agent has done across its entire session.
  • Multi-agent coordination: When multiple agents share a proxy, you get centralized visibility and can enforce cross-agent policies (e.g., "no two agents should modify the same file simultaneously").
  • Audit and compliance: The proxy's logs provide a complete audit trail for every AI interaction, useful for both debugging and regulatory compliance.

Conclusion

The proxy-over-API approach to AI control is practical, effective, and underexplored. It provides the full visibility and real-time enforcement that other approaches lack, with minimal setup cost and no dependency on model providers.

As AI agents become more autonomous, the gap between "what the model was trained to do" and "what we need it to do in production" will only grow. A control proxy bridges that gap—not by making the model smarter, but by ensuring that someone is watching.

We're building this at Luthien. It's open source, early stage, and we'd love feedback from anyone working on AI control.
